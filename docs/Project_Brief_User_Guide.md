# Project Brief User Guide

## Overview

The Project Brief is a comprehensive document that provides a firm foundation for project initiation. It links the Project Mandate to the detailed Business Case and serves as the primary reference document for the project team.

## Getting Started

### Prerequisites

- You must be assigned to a project as Project Manager or have PMO Admin permissions
- The project must exist in the system
- An approved Project Mandate is recommended (but not required)

### Accessing Project Briefs

1. Navigate to your project
2. Click on "Project Brief" in the project menu
3. If no brief exists, you'll see an option to "Create Brief"

## Creating a Project Brief

### Method 1: Create from Mandate (Recommended)

If your project has an approved mandate:

1. Go to the Project Mandate view
2. Click "Create Brief from Mandate"
3. The brief will be auto-populated with mandate information
4. Review and complete all sections

### Method 2: Create from Scratch

1. Navigate to your project
2. Click "Create Brief"
3. Fill in all required sections (marked with *)
4. Save as draft frequently

### Required Sections

The following sections are mandatory:

1. **Project Definition** - Background, objectives, and scope
2. **Outline Business Case** - Summary of business justification
3. **Product Description** - At least one product must be defined
4. **Project Approach** - Methodology and approach description
5. **Team Structure** - Team organization description
6. **Role Descriptions** - At minimum, Executive and Project Manager roles
7. **References** - Link to originating mandate (if applicable)

## Writing SMART Objectives

SMART objectives are:
- **Specific** - Clear and unambiguous
- **Measurable** - Has quantifiable metrics or targets
- **Achievable** - Realistic given available resources
- **Realistic** - Aligned with project constraints
- **Time-bound** - Has a clear deadline

### Example: Good SMART Objective

✅ **Good**: "Increase customer satisfaction scores from 75% to 90% by December 2024 through implementation of a new support ticketing system"

- Specific: Clear about what needs to be achieved
- Measurable: 75% to 90% provides metrics
- Achievable: Reasonable improvement target
- Realistic: Aligned with project scope
- Time-bound: December 2024 deadline

### Example: Poor Objective

❌ **Poor**: "Make customers happier"

- Not specific: What does "happier" mean?
- Not measurable: No metrics provided
- Not time-bound: No deadline

### Using the SMART Checker

1. Enter your objective text
2. The system will analyze it in real-time
3. Review the feedback and suggestions
4. Improve your objective based on the recommendations

## Defining Products

### What is a Product?

A product is any deliverable, outcome, or result that the project will produce. Products can be:
- Physical items (hardware, documents)
- Services (training, support)
- Outcomes (process improvements, capability enhancements)

### Adding Products

1. Navigate to the "Products" tab
2. Click "Add Product"
3. Fill in:
   - **Product Name** (required)
   - **Product Description** (required)
   - **Purpose** - Why this product is needed
   - **Quality Criteria** (required) - How quality will be measured
   - **Composition** - What the product consists of
   - **Derivation** - What it's based on
   - **Quality Tolerance** - Acceptable quality range
   - **Quality Method** - How quality will be verified

4. Mark one product as "Main Product" if applicable

### Product Quality Criteria

Quality criteria should be:
- Specific and measurable
- Testable or verifiable
- Aligned with customer expectations
- Documented in acceptance criteria

Example:
- "All pages must load in under 2 seconds"
- "System must support 1000 concurrent users"
- "Documentation must pass technical review"

## Team Structure Design

### Defining Team Structure

1. Navigate to the "Team Structure" tab
2. Describe the overall team organization
3. Include:
   - Reporting lines
   - Team composition
   - Organizational structure
   - Key relationships

### Adding Roles

1. Navigate to the "Role Descriptions" tab
2. Click "Add Role"
3. Fill in role details:
   - **Role Name** (required)
   - **Role Category** (Executive, Project Manager, Team Manager, etc.)
   - **Role Description**
   - **Key Responsibilities**
   - **Reporting To** - Who this role reports to
   - **Assigned To** - User assigned to this role (optional)
   - **Time Commitment** - Expected time allocation
   - **Mandatory** - Check if this role is required

### Mandatory Roles

The following roles are typically mandatory:
- **Executive** - Project sponsor/executive
- **Project Manager** - Day-to-day project management

### Team Structure Chart

The system automatically generates a visual team structure chart based on:
- Role definitions
- Reporting relationships
- Role categories

View the chart in the Team Structure section after adding roles.

## PMO Approval Process

### Submission Process

1. Complete all required sections
2. Review the completion progress indicator
3. Check quality criteria checklist
4. Click "Submit for Approval"
5. Select approvers (if you have permission)
6. Add any notes or comments

### Approval States

- **Draft** - Work in progress, can be edited
- **Under Review** - Submitted, awaiting approval
- **Approved** - Approved, cannot be edited (new version required)
- **Rejected** - Rejected, can be edited and resubmitted
- **Archived** - Archived, read-only

### What Approvers Look For

Approvers will review:
- Completeness of all sections
- Quality of objectives (SMART criteria)
- Alignment with mandate
- Quality criteria completeness
- Team structure appropriateness
- Business case justification

### After Approval

Once approved:
- Brief becomes read-only
- Can be linked to Business Case creation
- Can be exported and distributed
- Version history is maintained

## Exporting and Sharing

### Export Options

1. **PDF Export**
   - Click "Export PDF" or "Print"
   - Use browser's print dialog
   - Save as PDF

2. **Word Export**
   - Click "Export Word"
   - Document will download as .doc file
   - Compatible with Microsoft Word

3. **Print View**
   - Click "Print"
   - Optimized for printing
   - Includes all sections formatted for paper

### Distribution List

1. Navigate to the brief view
2. Scroll to "Distribution List" section
3. Click "Add Recipient"
4. Select user or enter external recipient
5. Specify version distributed
6. Recipients will appear in the distribution list

### Sharing Best Practices

- Always share the latest approved version
- Include brief reference number
- Specify which version is being shared
- Use distribution list for tracking

## Tips and Best Practices

### Completeness

- Use the completion progress indicator to track your progress
- Aim for 100% completion before submission
- Review quality criteria checklist

### Quality

- Write clear, concise descriptions
- Use bullet points for lists
- Include specific examples where helpful
- Review for typos and clarity

### Collaboration

- Save drafts frequently (auto-save is enabled)
- Share draft with team for feedback
- Use comments/notes for questions
- Coordinate with stakeholders

### Version Control

- Each submission creates a new version
- Version history is maintained
- Previous versions remain accessible
- Reference specific versions when needed

## Troubleshooting

### Can't Edit Brief

- Check if brief is approved (requires new version)
- Verify you have edit permissions
- Ensure brief is not archived

### Missing Sections

- Check completion progress indicator
- Review required fields (marked with *)
- Ensure all mandatory sections are filled

### Export Issues

- Try a different browser
- Check browser print settings
- Ensure brief is saved before exporting

### Quality Criteria Not Met

- Review quality criteria checklist
- Address each failed criterion
- Check SMART objective validation
- Review mandate alignment
