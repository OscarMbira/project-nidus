# Scrum Events User Guide

## Overview

Scrum Events are time-boxed ceremonies that provide structure and regularity to Scrum teams. This guide covers Daily Scrum, Sprint Review, and Sprint Retrospective events, their purpose, how to conduct them, and best practices.

## Daily Scrum

### Purpose

The Daily Scrum (also called Daily Standup) is a 15-minute time-boxed event for the Development Team to synchronize activities and create a plan for the next 24 hours.

### When to Conduct

- **Frequency**: Every day at the same time
- **Duration**: 15 minutes maximum
- **Time**: Usually at the start of the workday
- **Participants**: Development Team (required), Scrum Master (facilitator), Product Owner (optional)

### Accessing Daily Scrum

1. Navigate to your project
2. Go to **Scrum** > **Daily Scrum**
3. Select the current sprint
4. View today's standup

### The Three Questions

Each team member answers:

1. **What did I do yesterday?**
   - Completed work
   - Progress made
   - Items moved to Done

2. **What will I do today?**
   - Planned work
   - Tasks to start
   - Goals for the day

3. **Are there any blockers?**
   - Impediments
   - Dependencies
   - Issues preventing progress

### Recording Standup

1. **Select Your Name**: Click on your team member name
2. **Answer Questions**: Fill in each question
3. **Add Blockers**: List any impediments
4. **Save**: Click **Save Standup**
5. **View Team Updates**: See what others shared

### Timer Feature

- **15-Minute Timer**: Automatic countdown
- **Start Timer**: Begin when standup starts
- **Time Reminder**: Alerts when time is running out
- **Keep Focused**: Helps maintain time-box

### Blockers Management

**Adding Blockers**:
1. During standup, identify blockers
2. Click **Add Blocker**
3. Describe the blocker
4. Set priority (High, Medium, Low)
5. Assign owner if needed

**Resolving Blockers**:
1. View blocker list
2. Click on blocker
3. Update status
4. Add resolution notes
5. Mark as resolved

**Blocker Escalation**:
- High-priority blockers escalate to Scrum Master
- Blockers may become impediments
- Track blockers in sprint retrospective

### Best Practices

1. **Be Concise**: Keep answers brief and focused
2. **Stay Standing**: Physical standup encourages brevity
3. **Start on Time**: Respect the time-box
4. **Focus on Work**: Discuss work, not problems (save for after)
5. **Update Board**: Keep sprint board current

### Common Mistakes to Avoid

- ❌ Turning it into a status report for managers
- ❌ Solving problems during standup (schedule separate meeting)
- ❌ Going over 15 minutes
- ❌ Not updating the sprint board
- ❌ Skipping standup when busy

## Sprint Review

### Purpose

The Sprint Review is held at the end of the Sprint to inspect the Increment and adapt the Product Backlog if needed. It's an opportunity for stakeholders to provide feedback.

### When to Conduct

- **Frequency**: Once per sprint (at sprint end)
- **Duration**: 1-4 hours (depending on sprint length)
- **Participants**: Development Team, Product Owner, Stakeholders, Scrum Master

### Accessing Sprint Review

1. Navigate to your project
2. Go to **Scrum** > **Sprint Review**
3. Select the completed sprint
4. View review dashboard

### Preparing for Review

**Before the Review**:
1. **Complete Work**: Ensure all done items are truly done
2. **Prepare Demos**: Set up demonstrations
3. **Invite Stakeholders**: Send calendar invites
4. **Review Sprint Goal**: Check goal achievement
5. **Prepare Metrics**: Gather sprint metrics

**Review Checklist**:
- [ ] All done items reviewed
- [ ] Demos prepared and tested
- [ ] Stakeholders invited
- [ ] Sprint metrics ready
- [ ] Product backlog updated

### Conducting the Review

**Review Structure**:

1. **Sprint Summary** (10-15 min)
   - Sprint goal achievement
   - Completed items overview
   - Velocity and metrics

2. **Demo Completed Work** (30-60 min)
   - Demonstrate each completed feature
   - Show working software
   - Highlight key achievements

3. **Stakeholder Feedback** (20-30 min)
   - Collect feedback on demos
   - Discuss priorities
   - Identify new requirements

4. **Product Backlog Update** (15-20 min)
   - Add new items based on feedback
   - Reprioritize backlog
   - Update estimates

### Recording Feedback

1. **Add Feedback Item**:
   - Click **Add Feedback**
   - Enter feedback description
   - Categorize (Positive, Improvement, Bug, New Feature)
   - Link to related backlog item if applicable

2. **Feedback Categories**:
   - **Positive**: What stakeholders liked
   - **Improvement**: Suggestions for enhancement
   - **Bug**: Issues found during demo
   - **New Feature**: New requirements identified

3. **Action Items**:
   - Convert feedback to backlog items
   - Prioritize based on feedback
   - Assign to future sprints

### Demo Checklist

**For Each Feature**:
- [ ] Feature works as expected
- [ ] Acceptance criteria met
- [ ] User story demonstrated
- [ ] Questions answered
- [ ] Feedback collected

### Best Practices

1. **Show Working Software**: Demo real features, not slides
2. **Keep It Interactive**: Encourage questions and feedback
3. **Focus on Value**: Emphasize business value delivered
4. **Be Honest**: Show what wasn't completed too
5. **Capture Feedback**: Document all feedback for backlog

### Common Mistakes to Avoid

- ❌ Only showing completed work (show incomplete too)
- ❌ Using slides instead of working software
- ❌ Not inviting stakeholders
- ❌ Ignoring feedback
- ❌ Not updating product backlog

## Sprint Retrospective

### Purpose

The Sprint Retrospective is an opportunity for the team to inspect itself and create a plan for improvements to be enacted during the next Sprint.

### When to Conduct

- **Frequency**: Once per sprint (at sprint end, after review)
- **Duration**: 1-3 hours (depending on sprint length)
- **Participants**: Development Team, Scrum Master (required), Product Owner (optional)

### Accessing Retrospective

1. Navigate to your project
2. Go to **Scrum** > **Sprint Retrospective**
3. Select the completed sprint
4. View retro board

### Retrospective Format

**Three Categories**:

1. **What Went Well**
   - Positive aspects of the sprint
   - Successful practices
   - Achievements to celebrate
   - Things to continue doing

2. **What to Improve**
   - Challenges faced
   - Areas for improvement
   - Problems encountered
   - Things to change

3. **Action Items**
   - Concrete improvements
   - Specific next steps
   - Assigned owners
   - Due dates

### Participating in Retro

1. **Add Items**:
   - Click **Add Item** in relevant category
   - Enter your thought
   - Be specific and constructive

2. **Vote on Items**:
   - Vote on important items
   - Use voting to prioritize
   - Focus discussion on top-voted items

3. **Discuss as Team**:
   - Discuss each category
   - Share perspectives
   - Identify root causes
   - Brainstorm solutions

4. **Create Action Items**:
   - Convert improvements to actions
   - Assign owners
   - Set due dates
   - Track in next sprint

### Retrospective Techniques

**Start, Stop, Continue**:
- **Start**: New things to begin doing
- **Stop**: Things to stop doing
- **Continue**: Things to keep doing

**Mad, Sad, Glad**:
- **Mad**: Frustrations and irritations
- **Sad**: Disappointments and concerns
- **Glad**: Positive experiences

**4Ls (Liked, Learned, Lacked, Longed For)**:
- **Liked**: What went well
- **Learned**: New insights
- **Lacked**: What was missing
- **Longed For**: What was desired

### Action Items Management

**Creating Action Items**:
1. From improvement items, create actions
2. Be specific and measurable
3. Assign owner
4. Set due date (usually next sprint)
5. Track progress

**Following Up**:
- Review action items in next retro
- Check completion status
- Celebrate completed actions
- Adjust if needed

### Best Practices

1. **Safe Environment**: Create psychological safety
2. **Focus on Process**: Not on individuals
3. **Be Constructive**: Offer solutions, not just problems
4. **Limit Actions**: 1-3 action items per sprint
5. **Follow Through**: Actually implement improvements

### Common Mistakes to Avoid

- ❌ Blaming individuals
- ❌ Creating too many action items
- ❌ Not following up on previous actions
- ❌ Skipping retrospectives
- ❌ Not creating a safe environment

## Integration with Sprint Board

### Linking Events to Board

- **Daily Scrum**: Updates reflected on sprint board
- **Sprint Review**: Done items reviewed from board
- **Retrospective**: Board issues discussed in retro

### Workflow

1. **During Sprint**: Daily Scrum updates board
2. **Sprint End**: Review completed board items
3. **After Review**: Retrospective on sprint process
4. **Next Sprint**: Apply learnings to new sprint

## Tips for Scrum Masters

### Facilitating Events

1. **Daily Scrum**: Keep it focused and on time
2. **Sprint Review**: Ensure stakeholders attend
3. **Retrospective**: Create safe space for discussion

### Common Challenges

- **Standup Too Long**: Enforce time-box strictly
- **Low Attendance**: Make events valuable
- **No Feedback**: Ask specific questions
- **No Action Items**: Help team identify concrete actions
